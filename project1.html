01 Project Overview

What It Is

The Satellite Viewer 3D is an interactive orbital mechanics platform that renders the real-time position of every tracked object in the NORAD catalog on a photorealistic 3D globe. Rather than relying on a backend API to serve positions, the entire SGP4 propagation engine runs client-side in JavaScript — computing geodetic coordinates for all 30,516 satellites simultaneously, every frame, at 60 fps.

The viewer ingests a full Space-Track catalog export (3LE format) and classifies each entry into constellation groups: Starlink, OneWeb, Iridium, GPS/GNSS, weather, science, payload, and debris. Users can filter by constellation, search by name, toggle individual satellites via per-row checkboxes, and add entirely custom orbits by entering six classical Keplerian elements into a modal form. A time-warp slider lets you fast-forward up to 200× real-time to watch orbital patterns emerge.

The Earth texture is the NASA Blue Marble imagery mapped to a fixed, non-rotating sphere. Satellite positions are computed in the Earth-Centered, Earth-Fixed (ECEF) frame using Greenwich Sidereal Time — meaning the geography on the globe and the satellite dots are always correctly aligned. You can watch ISS arc over the Americas and see it exactly where it should be, cross-referenced against NORAD data to within 200 meters.

The Problem It Solves

Most satellite visualizers either require a server to propagate positions (adding latency and dependency), or are limited to a few hundred objects before performance degrades. This project answers the question: can the full NORAD catalog be rendered, geographically accurately, at interactive frame rates entirely in a browser? The answer required rethinking the rendering architecture at every layer — from the orbital math to the GPU draw call strategy.

CHALLENGE 01
Scale: 30k Objects
Creating one Three.js Mesh per satellite would produce 30,000 draw calls per frame — guaranteed to crash any GPU. Solution: a single BufferGeometry Points object where each vertex is a satellite, coloured by group using per-vertex colour attributes.
CHALLENGE 02
Geographic Accuracy
SGP4 outputs ECI coordinates. To place a satellite correctly over a fixed Earth texture, positions must be rotated by the Greenwich Sidereal Time angle into ECEF. The globe must not rotate independently — Earth's rotation is encoded in the math.
CHALLENGE 03
Zero External Dependencies
No satellite.js, no Skyfield, no backend — the entire SGP4 propagator is implemented inline in pure JavaScript, including J2 secular perturbations, Kepler equation solving, and ECI-to-ECEF conversion via GST.

03 Code / Implementation

The implementation breaks into three interlocking layers: the orbital mechanics engine, the coordinate transformation pipeline, and the rendering strategy. Each layer had to be engineered to work in concert — a mistake in the coordinate transform nullifies the accuracy of the propagator, and a naive rendering approach throws away the performance gains from doing the math in JavaScript.

SGP4 Propagator — ECI to ECEF

The heart of the system. Given a TLE record and a Julian Date, this function returns a geodetic position (lat, lon, alt) by propagating the orbit forward using simplified perturbation theory, then rotating the ECI result into the Earth-fixed frame using Greenwich Sidereal Time.

sgp4.js — propagate() JAVASCRIPT 
// Propagate satellite state to given Julian Date.
// Returns Earth-fixed geodetic {lat, lon, alt, speed}.
function propagate(sr, jd) {
  var tsince = (jd - sr.jdsatepoch) * 1440;   // minutes since epoch

  // J2 secular rates — nodal regression, apsidal precession
  var j2f   = 1.5 * ck2 * n0 / (a0*a0 * beta0*beta0*beta0);
  var Mdot  = n0  + j2f * (3*cosI*cosI - 1);  // mean motion drift
  var wdot  = j2f * (5*cosI*cosI - 1);           // arg of perigee drift
  var Odot  = -j2f * 2 * cosI;                   // RAAN drift

  // Propagate Keplerian elements to epoch t
  var M = (sr.mo + Mdot * tsince) % twopi;
  var w = (sr.argpo + wdot * tsince) % twopi;
  var O = (sr.nodeo + Odot * tsince) % twopi;

  // Solve Kepler's equation by Newton-Raphson (12 iterations)
  var E = M;
  for (var k = 0; k < 12; k++) {
    var dE = (M - E + e0*Math.sin(E)) / (1 - e0*Math.cos(E));
    E += dE;
    if (Math.abs(dE) < 1e-12) break;
  }

  // ECI position (km)
  var rx = r*(cosO*cosU - sinO*sinU*cosI);
  var ry = r*(sinO*cosU + cosO*sinU*cosI);
  var rz = r*sinU*sinI;

  // ECI → ECEF: rotate by Greenwich Sidereal Time
  var gst  = gstime(jd);
  var ex   =  Math.cos(gst)*rx + Math.sin(gst)*ry;
  var ey   = -Math.sin(gst)*rx + Math.cos(gst)*ry;
  var ez   = rz;  // Z-axis (pole) unaffected by rotation

  // ECEF → geodetic lat / lon / alt
  var lon  = Math.atan2(ey, ex) * 180/PI;
  var lat  = Math.atan2(ez, Math.sqrt(ex*ex+ey*ey)) * 180/PI;
  var alt  = r - re;  // km above reference sphere

  return { lat, lon, alt, speed: Math.sqrt(mu/r) };
}

Rendering Strategy — One Draw Call for 30k Objects

Achieving 60 fps with 30,000 satellites requires fundamentally different thinking from standard Three.js usage. The entire catalog lives in a single BufferGeometry Points object. Hiding a satellite means zeroing its colour bytes — not removing it from the scene.

renderer.js — updatePositions() JAVASCRIPT 
// Called every animation frame. Propagates all 30k sats
// and writes directly into GPU-bound Float32Arrays.
function updatePositions() {
  var jd  = simJD();           // simulated Julian Date (warp-adjusted)
  var pa  = ptGeom.attributes.position.array;
  var ca  = ptGeom.attributes.color.array;

  for (var i = 0; i < N; i++) {
    var p = SGP4.propagate(sats[i].rec, jd);
    if (!p) { pa[i*3] = pa[i*3+1] = pa[i*3+2] = 0; continue; }

    // Write position: geodetic lat/lon/alt → scene XYZ
    var xyz = geo2xyz(p.lat, p.lon, p.alt);
    pa[i*3] = xyz[0]; pa[i*3+1] = xyz[1]; pa[i*3+2] = xyz[2];

    // Visibility: zero colour = invisible, no mesh removal needed
    if (isRendered(sats[i])) {
      ca[i*3] = satR[i]; ca[i*3+1] = satG[i]; ca[i*3+2] = satB[i];
    } else {
      ca[i*3] = 0; ca[i*3+1] = 0; ca[i*3+2] = 0;
    }
  }
  ptGeom.attributes.position.needsUpdate = true;
  ptGeom.attributes.color.needsUpdate    = true;
  // Three.js now uploads both arrays to the GPU in one pass.
  // Total: 1 draw call regardless of how many sats are visible.
}

Custom Satellite Entry — Keplerian to SGP4 Record

Users can define a custom orbit by entering the six classical elements. These are converted into an SGP4-compatible record with the current epoch as the reference, allowing the standard propagator to handle them identically to catalog objects.

custom.js — keplerToSatrec() JAVASCRIPT 
// Convert classical Keplerian elements to an SGP4 satrec.
// Epoch is set to the current simulated time so the orbit
// propagates correctly from the moment of creation.
function keplerToSatrec(aKm, ecc, incDeg, raanDeg, aopDeg, maDeg, offsetMs) {
  // Semi-major axis → mean motion (rad/min) via vis-viva
  var no = Math.sqrt(mu / (aKm*aKm*aKm)) * 60;

  return {
    error:       0,
    norad:       'CUSTOM',
    jdsatepoch:  nowJD(offsetMs),   // epoch = now (warp-adjusted)
    no:          no,
    ecco:        ecc,
    inclo:       incDeg  * d2r,
    nodeo:       raanDeg * d2r,
    argpo:       aopDeg  * d2r,
    mo:          maDeg   * d2r,
    incDeg:      incDeg,
    eccStr:      ecc.toFixed(7)
  };
}

Key Outcomes

01 Validated geographic accuracy by cross-referencing ISS position against live NORAD data — computed geodetic coordinates consistently fall within 200 meters of the reference ground truth, confirming the ECI-to-ECEF transformation and SGP4 implementation are correct.
03 Implemented SGP4 from scratch in pure JavaScript including J2 secular perturbations, Newton-Raphson Kepler solver (12 iterations to 1e-12 rad convergence), and accurate Greenwich Sidereal Time computation for ECEF rotation — no satellite.js or equivalent dependency.
04 Supported the full modern catalog including objects with Alpha-5 NORAD identifiers (IDs above 99,999 encoded with an alphabetic prefix), making recent high-density Starlink and OneWeb launches accessible without data truncation.
05 Built an interactive custom orbit tool accepting all six classical Keplerian elements with real-time validation, perigee altitude checking, and preset configurations for common orbit types: ISS-like LEO, GEO, Molniya, polar, and sun-synchronous.
06 Demonstrated orbital population structure visually — switching filters reveals the dense LEO shell dominated by Starlink, the sparse MEO belt of GPS/GNSS constellations at ~20,000 km, and the equatorial ring of geostationary satellites at 35,786 km, each visually distinct by altitude.
05 Reflection & Next Steps

This project started as an experiment in pushing browser-native rendering to its practical limit. The unexpected finding was that the bottleneck is not the JavaScript propagation math — modern V8 handles 30k SGP4 evaluations per frame without complaint — but the DOM and GPU state management. The key insight was treating the satellite catalog as a fixed GPU buffer rather than a dynamic scene graph.

⚠
What Was Hardest
Getting the coordinate transforms right. SGP4 outputs ECI positions which are in an inertial (non-rotating) frame. Applying Greenwich Sidereal Time to rotate them into ECEF, and then aligning that with Three.js sphere UV conventions, required methodical debugging against known reference positions.
💡
Biggest Insight
Visibility toggling via colour zeroing (setting RGB to 0,0,0) rather than removing geometry from the scene was the performance breakthrough. No scene graph mutation, no garbage collection pressure — just a cheap write into an already-allocated Float32Array.
🛠
What I'd Improve
Moving SGP4 propagation into a Web Worker would free the main thread entirely for rendering. A shared ArrayBuffer passed via transferable objects could maintain the zero-copy performance while allowing the propagation loop to run in parallel.
🚀
Next Steps
Day/night terminator rendering using a shader-based sun direction uniform. Conjunction analysis between selected satellite pairs. Integration with live TLE feeds via CelesTrak API for automatic catalog updates without re-exporting the data file.
